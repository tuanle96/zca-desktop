# Mobile app (`apps/mobile`) — Tauri iOS

A **Tauri Mobile** thin client that brings the cloud Zalo experience to iOS, at near-parity with
the desktop. It is a SvelteKit shell (Svelte 5 + Tailwind v4) over a small Rust core that reuses the
shared crates/packages — no `rusqlite`, no `zca-rust` on device.

## Architecture

- **Shared, not duplicated:** the frontend reuses `@zca/core-client` (the `invoke()` cloud wrappers)
  and `@zca/types` (all display DTOs live there); the Rust core wraps `zca-cloud-client` +
  `zca-keychain` and registers the **same `cloud_*` IPC commands** as the desktop. The device token
  stays in the OS keychain, passed over IPC as the `__keychain__` sentinel — never in the webview.
- **Mobile-specific UI** (touch, stacked navigation — desktop is multi-pane):
  - `lib/nav.svelte.ts` — a tab + push/pop stack store (no router lib).
  - `lib/components/shell/*` — `app-shell` (router), `app-header` / `large-title` (iOS bars),
    `screen` (scroll container), `tab-bar` (HTML fallback bar).
  - `lib/session-store.svelte.ts` (+ `session-realtime.svelte.ts`, `session-hydrate.ts`,
    `session-helpers.ts`) — a **cloud-only** port of the desktop session store: restore on launch,
    realtime SSE + backoff reconnect, multi-account buckets, send/upload, hydrate. No OS
    notifications, no local-IPC branches.
  - Screens: `components/{login,conversations,thread,contacts,account,settings}/*`.
- **Theme:** `lib/theme.svelte.ts` (light/dark/system, toggles `.dark`); tokens mirror the desktop
  in `src/app.css`.

## iOS dev workflow

```bash
cd apps/mobile
bun run check                          # type-check
bun run tauri ios dev "iPhone 17"      # Vite dev server + HMR; serves the webview live
xcrun simctl io <udid> screenshot /tmp/x.png   # observe
```

**Gotchas (these have bitten us):**
- **Use `tauri ios dev` for UI iteration** — it serves from the Vite dev server (HMR), so frontend
  edits hot-reload. `tauri ios build` (one-shot) *embeds stale frontend assets* when only the
  frontend changed — the installed app shows old UI.
- **Do not build via XcodeBuildMCP / raw `xcodebuild`** — the "Build Rust Code" phase calls
  `tauri ios xcode-script`, which panics `failed to read missing addr file …-server-addr` (only the
  parent `tauri ios build/dev` writes it). Go through the Tauri CLI.
- **`gen/apple` is gitignored** (`apps/mobile/.gitignore`) and regenerated by `tauri ios init`.
  Xcode build configs are lowercase `debug`/`release`.
- **Native / Info.plist changes** (URL schemes, native views) live in `gen/apple` and are generated
  by **xcodegen** from `project.yml`. `tauri ios dev` does **NOT** run xcodegen, so after editing
  `project.yml` or adding a `.mm`: `(cd gen/apple && xcodegen generate)` then restart `tauri ios dev`.
- **WKWebView JS `console.log`** reaches the unified log only after
  `xcrun simctl spawn <udid> defaults write app.zca.mobile WebKitDebugLogsPageMessagesToSystemConsoleEnabled -bool YES`
  (re-set after a fresh install); then `log stream --predicate 'eventMessage CONTAINS "[zca:"'`.

## Native iOS 26 Liquid Glass tab bar

Real Liquid Glass **cannot** be done in CSS in a WKWebView (WebKit lacks SVG-displacement
`backdrop-filter`; that technique is Chromium-only). So the tab bar is a **native `UITabBar`**
(which adopts Liquid Glass automatically on the iOS 26 SDK), injected into tao's window at runtime by
`gen/apple/Sources/zca-mobile/glass-tabbar.mm` and bridged to the web nav:

- native tap → `webView.evaluateJavaScript("window.__setTab('…')")`
- web → native → `window.webkit.messageHandlers.glassTabBar.postMessage({hidden, activeTab, unread})`
  (hide on a pushed thread, sync the active tab + unread badge)
- the HTML `<TabBar/>` is a graceful fallback, hidden once native signals `window.__onGlassTabBar()`.

Reusable technique: the **`tauri-ios-native-glass`** skill (`~/.claude/skills/`).

## OAuth auto-return (deep link)

The mobile login passes `platform=mobile` to OAuth start; the server (`apps/server`, migration
`202606080001_oauth_login_platform`) returns a `zca://verify?code=…` landing for mobile instead of
the desktop loopback. The app registers the `zca://` scheme (`project.yml` `CFBundleURLTypes`) and
`lib/deeplink.ts` (`@tauri-apps/plugin-deep-link`) routes it to `session.linkViaOAuthCode`. Requires
the server to be deployed. (Universal Links / `link.zca.app` need AASA + a real device — custom
scheme is used because it works in the Simulator.)
